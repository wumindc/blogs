# 进阶 Prompt 策略：为什么 Few-Shot、CoT 和结构化输出能让结果更稳

> 很多人学 Prompt，学到最后会陷入一种很微妙的幻觉：好像只要把话说得更完整、更“像专家”、更像一套咒语，模型就会自动变得可靠。
>
> 真实工程里，事情远没有这么浪漫。多数时候，模型出错不是因为你不够会“写提示词文学”，而是因为你把一个本来需要示例、推导、校验和系统分工的问题，粗暴地压成了一段自然语言命令。

我越来越倾向于把进阶 Prompt 看成一门“输入建模”的工作，而不是一门“措辞美化”的技巧。

所谓进阶，不是让 Prompt 变长，而是让模型在更好的输入结构里工作：

- 它需要什么示例，才能明白你真正的判断口径；
- 它需要怎样的中间步骤，才能别在复杂问题上跳步；
- 它需要什么输出边界，才能让系统接得住；
- 它什么时候根本不该继续靠 Prompt 硬扛，而该交给检索、工具或工作流。

::: info 这篇文章会回答什么
- Few-Shot 为什么常常比“写十条规则”更有效
- Chain of Thought 到底帮了什么忙，又在哪些场景会帮倒忙
- 为什么结构化输出几乎是 Prompt 工程进入生产环境的分水岭
- 如何判断一个问题该继续打磨 Prompt，还是该换成 RAG、Tool Use 或多阶段编排
:::

## 1. 先说结论：大多数 Prompt 问题，本质上不是“不会说话”，而是“不会建模”

举一个很常见的业务场景。

假设你要做一个内部审批助手，目标不是闲聊，而是让它判断“员工离职时，未休年假折算是否符合制度”。这类问题看起来像问答，实际上同时包含了四层难点：

1. 规则本身有多条前置条件；
2. 一部分条件来自制度文本，一部分来自员工实时信息；
3. 结果必须给出理由，而不是只给一个“是 / 否”；
4. 输出最后要被系统消费，最好还能形成结构化记录。

如果你只是写一句：

```text
请根据公司制度判断这名员工是否可以折算年假，并给出准确结论。
```

模型当然也会回答，但这种回答往往有两个毛病：

- 它会把“听起来合理”误当成“经过验证”；
- 它没有被迫暴露自己的判断路径，所以你很难知道它错在了哪一步。

这时候继续加形容词，比如“请严谨、专业、认真、全面地分析”，收益通常已经很有限。因为真正缺的不是语气，而是三样东西：

- **示例**：告诉它什么叫“符合你们组织口径的判断”；
- **步骤**：告诉它哪些条件必须先拆出来检查；
- **结构**：告诉它最终答案必须如何组织，方便系统验证。

这三样东西，分别对应 Few-Shot、CoT 和结构化输出。

## 2. Few-Shot 真正厉害的地方，不是“举例子”，而是“对齐判例”

很多人第一次接触 Few-Shot，会把它理解成“在 Prompt 前面多放几个样例”。这个说法没错，但不够准。

Few-Shot 真正有价值的地方，在于它像在给模型看“判例”。

你不是单纯告诉它规则，而是在告诉它：

- 什么样的输入值得注意；
- 哪些细节其实是关键；
- 你希望它如何组织判断过程；
- 在边界场景下，你们团队到底偏保守还是偏激进。

### 2.1 为什么 Few-Shot 常常比长篇规则更有效

因为自然语言规则很容易看起来明确，实际上却仍然留着大量解释空间。

例如你写：

```text
请识别用户是否属于“高风险流失客户”。
```

如果后面跟的是一堆抽象规则：

- 最近活跃度下降明显
- 投诉频次升高
- 价值贡献较高但满意度下降

这对人类分析师可能够用，但对模型来说，“明显”“较高”“下降”都还是模糊词。

而如果你给出 3 到 5 个高质量样例，事情会立刻不同。因为样例同时传递了：

| 维度 | 规则文本 | Few-Shot 示例 |
| --- | --- | --- |
| 概念边界 | 往往抽象 | 更具体 |
| 输出格式 | 需要额外说明 | 示例天然带格式 |
| 组织口径 | 容易歧义 | 能体现团队偏好 |
| 边界情况 | 难写清 | 可直接展示 |

换句话说，Few-Shot 不是在“补说明”，而是在“缩小解释空间”。

### 2.2 Few-Shot 最适合什么任务

在我看来，Few-Shot 最适合那些“答案不只看知识，还看口径”的任务。

例如：

- 情绪分类与标签判断
- 摘要风格统一
- 结构化抽取
- 风险分级
- 代码审查意见组织方式

这些任务的共同点是：模型不一定缺知识，但它缺少你这套业务或团队内部的判断习惯。

### 2.3 Few-Shot 最容易踩的坑

Few-Shot 之所以好用，也正因为它“教得太快”。教得快，就意味着教错也会错得很快。

最常见的几个问题是：

#### 示例太像“宣传海报”，不像真实输入

很多人写示例时，会忍不住把例子写得特别整洁。现实里的用户输入明明是脏的、短的、跳跃的，但样例却像教科书。这会导致模型在真实输入下失去方向感。

#### 示例只展示常规情况，不展示边界情况

如果 5 个示例全是最标准的 happy path，模型在遇到模棱两可的情况时仍然会乱猜。真正有价值的示例，往往不是“最典型”的，而是“最容易出错的边缘样本”。

#### 示例的输出风格和目标系统不一致

比如你最终希望得到 JSON，但 few-shot 给的却是自然语言段落；或者你最终希望得到很短的审计结论，但样例全是长篇分析。模型通常会学到你真正展示的东西，而不是你口头要求它做的东西。

### 2.4 一个更实用的 Few-Shot 原则

如果你要用 Few-Shot，不妨把样例当作“最小判例库”来写。每组样例至少问自己三个问题：

1. 这是不是现实里真的会出现的输入？
2. 它有没有覆盖容易分歧的边界？
3. 它的输出是不是已经接近我最终想上线的格式？

如果这三个问题都答不上来，样例再多也只是装饰。

## 3. CoT 不是让模型“显得聪明”，而是避免它在复杂问题上跳步

Chain of Thought（CoT）几乎是 Prompt 讨论里最容易被神化，也最容易被滥用的技术。

它真正解决的，不是所有任务，而是一类特定问题：

- 需要多步推理；
- 中间步骤彼此依赖；
- 如果直接跳到结论，很容易错得很自然。

2022 年那篇很经典的 CoT 论文讲得很直接：当模型能够先生成一串中间推理步骤时，它在算术、常识和符号推理等复杂任务上的表现会明显提升。[论文链接](https://arxiv.org/abs/2201.11903)

这件事为什么重要？

因为很多模型错误，并不是“完全不会”，而是“会一点，但跳步”。它太急于给出像答案的东西，于是中间本该显式检查的环节被压扁了。

### 3.1 一个非常典型的错误：模型不是不会算，而是不肯老老实实展开

假设你让模型判断：

- 员工入职 8 个月；
- 最近一年实际工作月数 7 个月；
- 年假按工龄分档；
- 离职时不足一年按比例折算；
- 但试用期期间某些天数不计入折算基数。

如果你让它直接给结论，它往往会做出一种很像有道理的错误：它并不会完全编造，而是在脑中把其中一两条规则省略掉。

你让它把步骤展开，本质上是在逼它：

1. 先列已知条件；
2. 再选适用规则；
3. 再做计算；
4. 最后再给结论。

这并不是“给模型加法术”，而是在减少它偷懒跳步的空间。

### 3.2 但 CoT 不是万金油

这里有个非常关键的现实判断：**不是所有任务都值得打开 CoT**。

如果你的任务只是：

- 简单分类
- 信息抽取
- 固定格式改写
- 基于明确证据的短结论

强行让模型“逐步思考”，通常会带来三个副作用：

- 响应更慢；
- Token 成本更高；
- 输出变长，反而更难接系统。

Anthropic 在最新的 Prompting Best Practices 里其实给了一个很有启发性的提醒：当模型已经具备较强 thinking 能力时，很多场景下更好的做法不是手写非常死板的推理步骤，而是给更一般化的高质量指导；过度提示反而会让模型“思考过度”，增加延迟和 token 消耗。[官方文档](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#leverage-thinking-and-interleaved-thinking-capabilities)

这句话背后的意思非常值得认真理解：

- 对旧模型或普通 GPT 型模型，显式 CoT 往往很有帮助；
- 对新一代 reasoning 模型，很多时候你需要的是“让它知道该认真推理”，而不是替它把每一步写死；
- 一旦系统对延迟和成本敏感，CoT 必须是有选择地开，而不是默认开。

### 3.3 OpenAI 和 Anthropic 官方文档其实都在提醒同一件事

OpenAI 现在的 Prompting 文档也明确提到，不同类型模型的最佳提示方式可能不同，复杂应用应该绑定具体模型快照，并且通过 evals 持续验证 Prompt 表现，而不是假设“这套 Prompt 永远成立”。[官方文档](https://developers.openai.com/api/docs/guides/prompting)

这意味着一条很现实的工程原则：

> CoT 不是一段可以四处复制的固定咒语，它是一种需要随着模型类型、成本目标和任务复杂度一起调整的策略。

### 3.4 一个更稳的 CoT 写法

我现在更推荐这样组织 CoT，而不是简单写一句 `think step by step`：

```text
请按以下顺序完成分析：
1. 提取已知事实，不做推断
2. 列出适用规则
3. 说明规则与事实如何对应
4. 给出结论
5. 如果信息不足，请明确指出缺失项
```

这种写法的好处有三个：

- 它不是抽象地要求“思考”，而是限定了思考的骨架；
- 它天然适合后续做结构化输出；
- 它让错误更容易暴露，你知道模型是事实提取错了，还是规则映射错了。

## 4. 自一致性不是“多问几次”，而是承认推理路径本身可能不稳定

CoT 让模型展开步骤，但还有一个问题没解决：**它展开的那一条路径，未必就是最靠谱的那一条。**

这就是 Self-Consistency 要解决的问题。

那篇论文的核心思路很漂亮：不要只拿一条贪心解码出来的推理链，而是采样出多条不同推理路径，最后选择最一致的答案。[论文链接](https://arxiv.org/abs/2203.11171)

这个思想真正打动人的地方在于，它承认了一件很现实的事：

> 大模型在复杂问题上，不一定缺“答案”，它更可能缺的是“稳定到同一个答案的路径”。

### 4.1 什么场景值得用自一致性

我会把它留给下面这些场景：

- 复杂但可验证的推理题；
- 多步计算；
- 成本比准确率次要的关键链路；
- 可以容忍多次采样，但不能容忍明显错判的任务。

### 4.2 什么场景不值得

如果你的任务是：

- 用户评论分类；
- 工单摘要；
- 标签抽取；
- FAQ 改写；

自一致性大多数时候不划算。因为：

- 你为了一点稳定性，付出了几倍调用成本；
- 延迟大幅上涨；
- 但任务本身并没有复杂到值得这样做。

它更像一个“高价值任务的可靠性增强器”，不是默认配置。

## 5. ToT 和 Prompt Chaining，真正解决的是“单轮上下文装不下完整工作流”

很多 Prompt 教程一讲高级策略，就喜欢一路推到 Tree of Thoughts（ToT）或者 Prompt Chaining，但经常忽略一个前提：这类方法真正有价值的地方，不是显得高级，而是承认**有些任务本来就不适合在一轮里做完**。

Anthropic 的文档里有一句我很认同的话：即便模型已经有较强的 thinking 能力，当你需要检查中间结果、强制经过某个固定流水线，或者明确分离多个子任务时，显式 prompt chaining 仍然很有用。[官方文档](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#chain-complex-prompts)

这句话其实击中了一个常见误区：

很多人把多阶段编排理解成“模型不够聪明才需要拆步骤”。恰恰相反，很多时候拆步骤不是为了弥补模型智力，而是为了：

- 可追踪；
- 可审计；
- 可测试；
- 可替换某一环；
- 可在失败后只重跑局部步骤。

### 5.1 Tree of Thoughts 什么时候真有意义

ToT 论文里展示的是一种更强的推理方式：模型不只沿着一条链往前，而是会探索多条可能路径，并根据中间评价选择继续走哪条。[论文链接](https://arxiv.org/abs/2305.10601)

它的确很有启发性，但你如果把它直接拿来套所有业务系统，很容易走偏。

因为多数企业任务并不是真的“搜索树问题”。它们更常见的困难是：

- 事实来源复杂；
- 步骤多；
- 每一步有不同系统参与；
- 最终输出需要被审查。

对这类场景来说，很多时候更合适的不是 ToT，而是更朴素的 Prompt Chaining：

1. 第一步先抽取事实；
2. 第二步根据事实生成判断草案；
3. 第三步做校验或重写；
4. 第四步再输出最终结果。

这种方式没有 ToT 那么“学术上炫”，但它更像真实业务系统。

## 6. 结构化输出，才是 Prompt 工程真正进入生产的门槛

如果你只把 Prompt 当成聊天技巧，那么 Few-Shot 和 CoT 已经足够让你惊艳一阵子了。

但如果你要把模型接进真实系统，事情会立刻变得没那么诗意。

系统不关心你写得多像人，它只关心三件事：

- 能不能解析；
- 能不能校验；
- 能不能接下去处理。

这就是为什么结构化输出如此关键。

### 6.1 工程里最常见的失败，不是模型“完全答错”，而是“半对半坏，系统吃不下”

比如你明明想要：

```json
{
  "risk_level": "high",
  "reasons": ["合同金额异常", "审批链不完整"],
  "need_human_review": true
}
```

模型却返回：

- JSON 前先写了一段解释；
- 枚举值写成了 `HIGH RISK`；
- `need_human_review` 写成了 `"yes"`；
- 或者干脆多加一个系统没定义的字段。

这类错误从阅读角度看不严重，但从系统角度看就是不可用。

### 6.2 结构化输出的价值，不只是“方便解析”

它真正改变的是你处理模型输出的方式：

- 可以用 Schema 做前置约束；
- 可以做字段级校验；
- 可以把失败变成可重试、可降级的系统分支；
- 可以把“模型输出”变成软件系统里的正式接口契约。

OpenAI 现在的文档里其实把这件事放得很前：Prompting 是一门艺术加科学，但如果要稳定落地，就应该尽量把结果放进可评估、可版本化、可复用的体系里，而不是让每次提示都散落在业务代码中。[官方文档](https://developers.openai.com/api/docs/guides/prompting)

### 6.3 一个常被忽略的点：结构化输出也会反过来提升思考质量

这件事很有意思。很多人以为结构化输出只是后处理需求，但实际用下来你会发现：

当你要求模型先输出明确字段，例如：

- `facts`
- `rules`
- `decision`
- `confidence`
- `missing_information`

模型本身的思路也会变清楚。因为你在强迫它把原本一团模糊的自然语言组织成几个功能清晰的槽位。

所以结构化输出并不只是为了机器，很多时候也是在帮模型整理自己。

## 7. 一个成熟团队怎么判断“该继续写 Prompt，还是该换系统方案”

这是我觉得最重要、也最容易被忽略的问题。

很多团队在模型表现不稳定时，本能反应都是“再改改 Prompt”。但 Prompt 的能力边界其实很明确。

### 7.1 适合继续打磨 Prompt 的情况

- 任务已经足够封闭；
- 你主要缺的是判断口径和输出稳定性；
- 结果不依赖外部实时数据；
- 错误主要来自表达混乱，而不是事实缺失。

这类问题，Few-Shot、CoT、结构化输出通常都很有效。

### 7.2 不适合继续硬打 Prompt 的情况

下面这些问题，如果还只靠 Prompt，基本就是在硬撑：

| 问题根源 | 更合适的解决方案 |
| --- | --- |
| 模型缺少事实 | RAG / File Search / 工具调用 |
| 需要实时数据 | Tool Use |
| 任务跨多阶段且有中间产物 | Prompt Chaining / 工作流编排 |
| 输出必须严格可用 | Structured Output + 校验 |
| 有高风险副作用 | 审批流 / 策略引擎 / 人工复核 |

最危险的一种情况是：问题明明是“没有事实”，团队却还在围着 Prompt 疯狂调措辞。最后得出的结论往往不是 Prompt 没用，而是“AI 不可靠”。其实不是 AI 不可靠，而是你把它放到了不属于 Prompt 的问题上。

## 8. 一个更像工程实践的落地模板

如果你要为一个真实业务任务设计进阶 Prompt，我建议至少按下面这套顺序思考：

### 8.1 先判断任务类型

- 这是分类、抽取、推理，还是规划？
- 它依赖外部事实吗？
- 最终结果是给人看，还是给系统吃？

### 8.2 再决定用什么策略

- 缺口径：加 Few-Shot
- 缺中间推导：加 CoT 或分步模板
- 缺稳定性：做自一致性或二次校验
- 缺可消费性：做结构化输出
- 缺事实：改 RAG 或 Tool Use

### 8.3 最后把 Prompt 当成“版本化资产”，而不是散落文本

这是上一轮我自己也做得不够好的地方。

Prompt 真正进入团队协作后，它不应该是一段“谁想到什么就改什么”的临时文本，而应该像代码一样：

- 有版本；
- 有评估；
- 有样例；
- 有回归测试；
- 有变更原因。

OpenAI 的 Prompting 文档现在也在强调这一点：Prompt 应该被版本化、可测试、可复用，而不是每次直接在请求里手搓一大段字符串。[官方文档](https://developers.openai.com/api/docs/guides/prompting)

## 9. 小结：Prompt 工程的成熟，不在于更会写，而在于更会分工

如果我现在重新概括这篇文章，我会把重点收束成一句话：

> 进阶 Prompt 不是把命令写得更像咒语，而是把一个问题拆成模型真正能吃下、系统真正能接住的输入结构。

Few-Shot 的价值，在于对齐判例和判断口径；  
CoT 的价值，在于让复杂问题别再偷懒跳步；  
自一致性的价值，在于承认推理路径并不总是稳定；  
结构化输出的价值，在于让模型输出从“像答案”变成“可用接口”。

而真正成熟的工程判断，是知道什么时候继续打磨 Prompt，什么时候干脆承认：这已经不是 Prompt 的工作，而该交给检索、工具和工作流了。
