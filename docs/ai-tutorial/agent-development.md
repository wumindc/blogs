# 赋予代码灵魂：智能体开发实战 (Function Calling)

> 上一章我们了解了 Agent 能够在自己的脑海里进行 ReAct 推理循环的雄伟蓝图。
> 那么作为一名开发人员，大模型到底是通过怎样的数据结构，向我们提出“我想调用你电脑里的那个 API 发邮件”这个疯狂企图的呢？
> 答案就是 OpenAI 引领的业界规范核心特性：**函数调用 (Function Calling)**。

---

## 1. 传统解析的血泪史 (Regex 噩梦)

在 Function Calling 时代还没到来之前，为了让 LLM (比如早期的 GPT-3.5) 能够触发我们的函数。我们要在 Prompt 里卑微地写下如同长篇论文般的规约：

> *“请你输出 JSON！并且只能输出 JSON！必须要包含 `methodName` 和 `args` 这两个属性。不要带任何 Markdown 的 ````json` 的语法壳子！不要在结尾说谢谢！”*

但是大模型是个概率机器，有时候它心情好完美输出，有时候它还是会在结尾加上一句“好的，我马上帮你调函数 `{"methodName": "xxx"}` ”。这会导致我们后端用 `JSON.parse` 去做反序列化时，频繁报错崩溃退出。

## 2. 划时代的官方支持：`tools` 参数

为了终结这种乱象调用的折磨，模型厂商在训练底座模型时，专门下发了结构化的微调。并在 `/v1/chat/completions` 这个核心 API 里新增了一个名为 `tools` 或 `functions` 的霸道字段。

它的核心理念极其简单：**你在请求对话的同一时间，顺便向大模型“注册”一份你当前后端手里握着的所有“武器清单 (`tools`)”。**

### 第 1 步：注册武器清单（工具的描述一定要极其精准）
当你发送 HTTP 请求时，你的体位变成了这样：

```json
{
  "model": "gpt-4o",
  "messages": [{"role": "user", "content": "帮我查一下杭州今天的天气然后发给老大"}],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "获取指定城市的当前真实天气情况，必须传入省市名称。",
        "parameters": {
          "type": "object",
          "properties": {
            "location": {
              "type": "string",
              "description": "城市名称，如：杭州市"
            }
          },
          "required": ["location"]
        }
      }
    },
    {
       // ... 第二个工具 (如 send_email) 的说明定义，以此类推
    }
  ],
  "tool_choice": "auto"   // 让大模型自己决定要不要调工具
}
```

### 第 2 步：大模型“挂起”通话，要求你替它执行 (tool_calls)
由于 `get_weather` 的描述和参数定义（基于标准的 JSON Schema 规范）写得极度详尽，大模型的注意力矩阵会瞬间判断由于用户在请求天气，它自己的脑子里不具备实时能力。

这次 API 响应，它**竟然一句话文本的寒暄都没跟你说**，而是返回了以下这个严酷的数据结构：

```json
{
  "choices": [
    {
      "finish_reason": "tool_calls", // 重点！！不再是 "stop" 结束
      "message": {
        "role": "assistant",
        "content": null,
        "tool_calls": [
           {
             "id": "call_abc123",
             "type": "function",
             "function": {
               "name": "get_weather",
               "arguments": "{\"location\": \"杭州市\"}"   // 它完美地将你的参数要求填充好了！
             }
           }
        ]
      }
    }
  ]
}
```

### 第 3 步：跑完函数，将观察结果塞回它的嘴里
此时我们的后端 Spring Boot / Node.js 代码根据 `tool_calls` 用一个大 `switch-case` 将 `杭州市` 作为形参输入进去，发起了真实的高德地图 API 查寻，得出字符串体结果：`"晴天，32度"`。

接着，我们把这句结果，连带着刚才那个 `call_abc123` 打包成一个 `role: "tool"` (专属的工具执行结果角色) 追加在刚才极其庞大的历史数组后面，重新发起第二次的 `/v1/chat/completions` API 调用。

它心满意足地吃下了真实数据，最终返回了一句漂亮的话：
> *老大，今天杭州是晴天，气温高达 32 度，记得少穿点。* （紧接着，它可能会在后台发出第二个要求 `送发邮件` 的 Function Call）

## 3. Function Calling 对工程架构的颠覆

当函数调用被结构化且变得极度稳定时，这意味着一切可以由人类用鼠标或命令行去点击操作的行为，它全都能接管。

*   **数据分析 Agent**: 你只需提供一个能够 `执行 Python 脚本代码` 的 Function 给它。遇到极其复杂的 Excel 提问时，它会疯狂地生成使用 Pandas 排序的分析代码，然后扔进你给它挂载的一个本地 Docker 沙盒环境里去帮你把分析跑出来并渲染制图。
*   **私有数据库 DBA**: 你只需挂载一个能执行 `执行任意 SQL 查询` 的只读 Function。你甚至不需要再去开发那些繁复的 ERP 图表系统，大模型直接把自然语言转成了 SQL 跑完结果并分析返回（Text-to-SQL）。
