# 基于 AI 的日常开发提效方案

> 真正的“效能飞跃”，并不是整天在极速打字，而是大幅减少花在阅读垃圾代码、寻找八股文文档、写繁杂样板代码上的低价值脑力消耗。 

有了大模型的帮助，我们在日常开发的几个高频环节中，可以总结出以下非常实用的“骚操作”。

---

## 场景 1：快速处理祖传的晦涩代码 (Legacy Code)

接手新项目或老兵遗留下的、完全没有注释、“屎山”一样的代码是开发者的家常便饭。现在有了 AI，你不再需要硬啃：

### 逆向分析
如果你看不懂这段数百行的业务核心算法：
> **技巧：** 框选晦涩的代码，在编辑器的 AI 侧边栏中输入：
> *"请按业务步骤，以序号列表的形式为我总结这段 Java 代码干了什么？这段代码里面有没有潜在的 NPE（空指针异常）风险或死循环？"*

### 增加行内注释
> **技巧：** 
> *"请逐行为这段函数加上详尽的中文注释，并且使用 Javadoc 的格式将参数含义标注在方法头上。"*
AI 不仅能翻译出语法本身，它常常能结合变量名词猜测出这段代在真实业务上的意图。

---

## 场景 2：那些我们懒得写的“脏活累活”

这类工作难度为 0，但极度无聊且极其费时。这正是 Token 机器人的最佳舞台：

### 复杂的正则表达式
人类不应该记忆正则。当你需要一个匹配特定格式的车牌号或者提取复杂文本中间的某种时间戳时。
> **示例 Prompt：** *"用 Python 写一个正则式：匹配一句话中出现的所有连续两位数字，但是这俩数字前面必须出现过'日期'这两个字。列举 3 个测试用例证明你的式子是对的。"*

### 写测试用例（Unit Test）
测试用例的覆盖其实全靠排列组合找边缘用例（Edge Cases）。
> **示例 Prompt：** *"为下面这个处理购物车满减逻辑的函数编写 JUnit5 / Mockito 单测。考虑：用户未登录、满金额刚好卡点、超出最大优惠界限这三个边缘异常场景。"*

### SQL 的转换与查漏
不管是对着长达数个 JOIN 语法的复杂 SQL 找由于忘记写别名引起的 Ambiguous Column 报错，还是说把 MySQL 语法一键转成 PostgreSQL 支持的语法，这对于 AI 都是“降维打击”。

---

## 场景 3：重构与规范化 (Refactoring)

代码能跑，不代表代码漂亮。当你需要合并同类项，或者调整老代码去适应你们公司的新规范时。

> **示例 Prompt：**
> *“这段代码有 5 层深的 `if/else` 嵌套，并且很多重复的长字符串赋值。请使用 Early Return（提前返回）策略将嵌套深度压平，并将其中的长硬编码抽取成顶部的一个统一 Enum 枚举类或常量对象。要求保持原本的逻辑不动。”*

这类重构任务在人工执行时非常容易因为漏掉了一个细小的条件分支改出线上 Bug，而大模型凭借着海量注意力的扫描，这种结构转换（不涉及大业务架构变更时）往往极其精准无瑕疵。

---

总的来说，日常应用大模型的精髓在于将它视为一个**不知疲倦、熟读计算机基础的实习生**。不要一上来就让它完成一个大项目，而是不停地把日常编码中微小的、需要搜索引擎去查文档的事拆出去让它代劳。

为了确保这位“实习生”能听懂你的指令，在它的眼里，不仅你的 Prompt 重要，**代码的上下文同样重要**。请移步：**[如何写出让 AI 秒懂的代码注释](./context-prompting)**。
