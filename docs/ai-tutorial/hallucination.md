# AI 幻觉的成因解析与缓解：从采样到系统防线

> 很多人第一次见识 AI 幻觉，都会有一种非常微妙的感受：它不是在随便乱说，而是在一本正经地乱说。
> 也正因为如此，幻觉才格外危险。它不像普通程序报错那样明显，它往往语言流畅、逻辑通顺，甚至还带着一点让人放松警惕的自信。要真正处理这个问题，靠的不是一句“请不要胡编”，而是先看清它到底是从哪一层冒出来的。

::: info 这篇文章重点
- 幻觉为什么是生成式模型的天然风险
- 温度、采样、知识缺失和上下文噪声如何影响幻觉
- 如何从 Prompt、检索、结构化校验和系统流程四层缓解风险
:::

## 1. 幻觉不是 bug，而是概率生成的副作用

大模型生成下一个 Token 的方式，本质上是在候选概率分布中做选择。只要系统不是直接检索事实库，而是在“根据上下文生成最可能的下一个词”，它就天然可能：

- 补全不存在的事实
- 编造看似合理的引用
- 给出语言流畅但证据不足的结论

所以幻觉不是一种特殊故障，而是生成式系统的结构性风险。

## 2. 幻觉最常见的四个来源

### 2.1 模型缺少关键事实

如果问题需要外部知识，而上下文里没有提供，模型仍然会倾向于给出一个“尽量像答案的答案”。

### 2.2 上下文噪声过多

当检索结果杂乱、候选太长、提示词不聚焦时，模型可能在噪声中抓住错误线索并继续扩写。

### 2.3 采样带来的随机性

更高的温度、更开放的采样空间，会增加表达多样性，也会放大不稳定性。

### 2.4 任务本身缺乏可验证标准

如果问题没有明确边界，比如“帮我全面判断这个合同是否存在一切风险”，模型很容易在大范围模糊任务中自信扩展。

## 3. 温度和 Top-p 的作用，不是简单的“越低越准”

### 3.1 Temperature

Temperature 越高，输出越发散；越低，输出越保守。

这通常意味着：

- 创意写作适合稍高温度
- 信息抽取、分类、结构化输出更适合低温度

但要注意，温度低不等于绝对正确。它只能减少随机性，不能凭空补足缺失的事实。

### 3.2 Top-p

Top-p 通过截断低概率候选来限制采样空间。它和温度一样，影响的是“怎么选”，而不是“事实从哪里来”。

所以当问题根源是知识缺失时，继续调采样参数通常治标不治本。

## 4. 生产系统里的四层防线

### 4.1 Prompt 层：让模型知道什么时候该保守

一个实用原则是明确告诉模型：

- 如果证据不足，要说“不确定”
- 不要伪造来源
- 如果需要外部事实，应优先使用提供的资料

这类提示不会消灭幻觉，但能明显改善“瞎编还很自信”的情况。

### 4.2 检索层：把事实先送到位

如果问题依赖企业知识或实时信息，最有效的方式通常不是继续打磨提示词，而是：

- 用 RAG 补足静态事实
- 用工具调用补足实时数据

很多看起来是“模型胡说八道”的问题，根本原因其实是系统没有提供足够事实。

### 4.3 输出层：把回答变成可校验结构

结构化输出能帮助你在系统层面识别很多问题：

- 字段缺失
- 枚举越界
- 数值格式错误
- 引用来源为空

这意味着即便模型没有完全“更聪明”，系统仍然可以拦下很多不可用结果。

### 4.4 流程层：把高风险结论从“自动执行”改成“待确认”

对于下面这些场景，不应该直接相信模型自然语言结论：

- 法务判断
- 医疗建议
- 财务动作
- 人事处罚

更稳的做法是把模型输出变成“待审核建议”，而不是“直接执行结果”。

## 5. 一个更实用的判断框架

如果你遇到幻觉问题，可以先问：

1. 是没有事实，还是没有正确召回事实？
2. 是输出太发散，还是任务本身定义模糊？
3. 是自然语言回答不可控，还是缺少结构化校验？
4. 是普通误差，还是已经进入高风险业务场景？

这四个问题能帮助你判断该调 Prompt、调检索、加校验，还是改业务流程。

## 6. 常见反模式

### 6.1 只靠一句“不要幻觉”

这种提示几乎没有系统性约束力。

### 6.2 把所有希望都寄托在低温度

低温度可以更稳，但不能创造事实。

### 6.3 不给引用来源

没有来源展示，用户和开发者都无法判断结论依据。

### 6.4 让模型在高风险场景直接落动作

这不是“大胆”，而是把不确定性直接传给业务系统。

## 7. 小结

幻觉不是一个可以靠单一技巧彻底消灭的问题。更现实的做法是分层治理：

- Prompt 层降低无依据扩写
- 检索和工具层补足事实
- 结构化输出层拦截不合规结果
- 业务流程层控制高风险动作

当你把幻觉看成系统问题，而不是模型性格问题，解决思路就会清晰得多。
