# 算力与硬件：显存墙、KV Cache 与 GPU 集群为什么这么重要

> 如果你把大模型简单理解成“一段更大的程序”，很多硬件现象都会看不懂。你会困惑：为什么卡这么贵，为什么一上并发就炸，为什么纸面算力已经很夸张了，推理速度还是不够体面。
> 把视角从应用层往下压一层，这些问题其实都不神秘。大模型更像一个被算力、带宽、显存和互联死死拽住的计算系统，而不是一段随便找台机器就能优雅跑起来的软件。

::: info 这篇文章重点
- 为什么 GPU 比 CPU 更适合大模型训练和推理
- 为什么推理瓶颈经常不是算力，而是显存带宽
- KV Cache 为什么会把并发成本迅速放大
- NVLink、IB 这类互联为什么会成为集群关键
:::

## 1. 大模型为什么偏爱 GPU

传统业务程序包含大量分支、条件跳转和复杂控制逻辑，这正是 CPU 擅长的领域。

而大模型的核心计算主要是：

- 矩阵乘法
- 向量加法
- 大规模并行数值计算

这类工作负载更适合 GPU 这种“并行吞吐优先”的架构。

## 2. 只看 FLOPS 会误判问题

很多人谈大模型硬件时，只盯着算力数字。但在推理阶段，真正常见的瓶颈是**内存带宽**。

原因很简单：

- 模型参数体积巨大
- 生成每个 Token 时都要频繁读取权重
- 计算本身未必比“搬数据”更慢

这就是常说的 **Memory Wall（显存墙）**。

## 3. 生成一个 Token 为什么这么贵

从推理角度看，生成一个 Token 不只是“算一个字”，而是反复执行：

1. 读取模型权重
2. 做一轮前向计算
3. 更新上下文相关状态
4. 再为下一个 Token 重复

因此在大模型推理中，“把权重从显存搬到计算单元”的成本经常非常显著。

## 4. KV Cache：并发一上来，显存就先爆了

除了模型权重，大模型推理还会消耗另一块关键资源：KV Cache。

KV Cache 可以理解成模型为了避免重复计算历史上下文而保存的中间状态。它的价值很大，但代价也明显：

- 上下文越长，缓存越大
- 并发请求越多，缓存总量越高
- 即使模型权重是共享的，KV Cache 也是请求级别占用

所以很多系统不是死在“模型装不下”，而是死在“模型能装下，但并发一起来缓存撑爆显存”。

## 5. 为什么推理引擎和调度策略很关键

到了这一层，问题就不只是“有没有卡”，而是：

- 如何管理显存
- 如何组织批处理
- 如何减少碎片
- 如何让不同请求共享更多资源

这也是为什么生产环境里通常不会用最原始的推理方式直接上线，而会引入专门的推理引擎和调度策略。

## 6. 多卡与多机：互联能力决定能不能扩展

当模型大到一张卡装不下，或者吞吐需求超过单机能力，就要进入多卡、多机场景。

这时新的问题变成：

- 卡间怎么通信
- 权重和激活怎么切分
- 跨机延迟能不能承受

因此 NVLink、NVSwitch、InfiniBand 这类高速互联会变得非常关键。算力再强，如果互联跟不上，整体吞吐也会被拖住。

## 小结

理解大模型硬件，不是为了背硬件型号，而是为了知道系统为什么会慢、为什么会贵、为什么并发一上来就不稳定。

对工程师来说，最值得建立的直觉是：

- GPU 不是因为“更高级”，而是更适合大规模并行数值计算
- 推理瓶颈常常是显存带宽，不只是算力
- KV Cache 会显著影响并发成本
- 多卡和多机扩展依赖高速互联
