# 拥抱 AI Native：接入大模型 API 核心指南

> 对于真正的开发者来说，在网页里玩对话机器人只是第一步。我们的终极目标是将这种巨大的“推理和生成算力”嵌入到我们自己构建的系统中。那么，**如何通过代码让服务器跟这些“最强大脑”沟通呢？**

在这一节，我们将了解大模型 API 的调用原理以及主流的平台。绝大多数现代大语言模型厂商，为了方便开发者，都向后兼容甚至像素级模仿了最经典的 **OpenAI Chat Completions API** 规范。这就意味着，只要你会调一次，以后可以随时用同样的代码切换各种国产或全球顶尖的大模型。

---

## 认识核心接口模型：Chat Completions
大模型对话目前最核心且通用的接口范式，叫 **Message 数组传递**。

模型本质上是没有“记忆”的（Stateless无状态）。不管你聊到第十句还是第一千句，当你向服务器发请求时，**你必须把之前的全量对话历史，原原本本地作为 JSON 数组发给大模型**。它看完历史数组后，再“接龙”生成新的一句。

### Message 的三种身份流（Role）

一个典型的发送给请求体（Request Payload）长这样：
```json
{
  "model": "gpt-4o",  // 指定你租用的是哪一个大脑
  "messages": [
    { "role": "system", "content": "你是一个严厉的 Java 架构师老师。" }, 
    { "role": "user", "content": "我的循环一直报错空指针，能帮忙看看吗？" },
    { "role": "assistant", "content": "把你报错的堆栈发出来，别废话。" },
    { "role": "user", "content": "这是堆栈：NullPointerException at xxx" }
  ],
  "temperature": 0.7  // 决定模型生成答案的“随机性水平”（0为极其严谨死板，1为脑洞大开胡乱联想）
}
```

* **System (系统角色)**：最高优先级的核心预设（即你设置 RCTF 中的 Role 和 Format 经常放在此处）。
* **User (用户角色)**：终端发出的提问（比如你在前端对话框打出的内容）。
* **Assistant (AI 助手角色)**：其实就是模型返回的回答，在下一轮对话时你要原封不动塞回数组里发过去。

## 主流大模型平台推荐

根据业务是否出海、合规性要求以及预算，开发时我们的选型面很广：

### 全球通用霸主：OpenAI / Anthropic (Claude)
- **特点**：最纯正的生态位，拥有世界上最强的闭源智能推理能力。
- **调用库推荐**：各自官方都提供了强大的 SDK： `npm install openai` 或者直接使用 `pip install openai`。如果你嫌麻烦，可以直接用像 **LangChain** 这种顶通生态库。
- **门槛点**：对于网络、支付门槛（海外信用卡）要求较高。

### 国内合规首选：百川、通义千问、Kimi (月之暗面)、智谱
- **特点**：国内节点极其快，中文特供优化，完全兼容国内互联网 ICP 审核。而且它们基本上都支持以“兼容 OpenAI SDK 格式”来调用。
- **费用**：现阶段极其便宜，很多模型比如 `DeepSeek` 或者 `Qwen`(千问) 都有着巨额免费额度，且价格比 API 大战前低了数个数量级。

### 开发机内私有化部署：Ollama
- **特点**：如果你手里有一台 16G 显存（或者MacBook M系列带内存共享架构）的机器，且不希望任何企业机密代码通过网络请求走出去。
- **用法**：安装 `Ollama` 之后，在终端通过 `ollama run llama3` 等命令直接把开源大模型下载到硬盘里并在本机内存里跑起来。同时 Ollama 默认开启了 `localhost:11434` 服务器，暴露出和 OpenAI 语法一模一样的本地 API。
