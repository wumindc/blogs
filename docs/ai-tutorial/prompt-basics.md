# Prompt 核心框架：给黑盒下达标准指令的艺术 (RCTF)

> 为什么有人用大模型像个神仙，写代码做企划无所不能；而有人用大模型，它总是回一句“作为人工智能我无法回答”或者直接给你一段驴唇不对马嘴的废话？
> 核心差距在于：**提示词工程 (Prompt Engineering)**。

由于我们在上一章讲过，模型本质上是一个接龙机器。你给的“前奏”越精确、给它的“语境”越厚重，它接龙接出来的准确度就越高。

业界最通用、最高效也是最不易翻车的一个基础范式，是 **RCTF 架构**。

## 1. R: Role (角色设定)
> **“你是一个拥有 20 年经验的高级 Java 架构师，精通高并发与分布式系统设计。”**

**原理解析**：
模型在预训练时，阅读过互联网上的小学生作文、民科言论，也阅读过 Linux 系统内核源码和顶级学术论文。
如果你直接问一个技术问题，它可能会在这些浩如烟海的数据中随机取样。但当你加上一个“高级架构师”的 Role 设定后，你在它的注意力机制（Self-Attention）中，瞬间激活了所有高质量代码库和顶级技术博客的权重域，过滤掉了那些低质量的废话。这叫作 **“通过扮演强制收敛上下文分布”**。

## 2. C: Context (背景上下文)
> **“我目前正在开发一个给医院使用的挂号排队系统。现在遇到一个问题，在早上 8 点放号时，经常会出现超卖（库存为负）的情况。我们的技术栈是 Spring Boot + Redis + MySQL。”**

**原理解析**：
大模型没有记忆。你不给它具体的业务背景，它只能给你百度百科级别的空洞鸡汤。
你给出的现有技术栈（Redis + MySQL）和业务场景（医院早高峰抢号），极大地收束了它的推理方向，阻止它胡乱建议你使用诸如 MongoDB 这种不符合现有团队技术栈的花架子。

## 3. T: Task (任务陈述)
> **“请帮我设计一段基于 Redis Lua 脚本的原子扣减库存方法逻辑，并用 Java 写出一个调用的 Demo。”**

**原理解析**：
任务必须**清晰、可衡量、具有唯一指向性**。
不要说：“帮我解决这个问题”。这太宽泛了，模型可能会直接扔给你一大段理论探讨，不给代码。明确指出“我要一段 Lua 脚本”和“一个 Java 调用 Demo”，让这个接龙机器有了极其明确的终止目标。

## 4. F: Format (输出格式限定)
> **“请按以下格式输出：
> 1. 先简单解释一下 Lua 脚本能防止超卖的原理（不超过 50 字）。
> 2. 输出 Lua 脚本代码。
> 3. 输出基于 Spring Data Redis 的 Java 方法（写明所有的依赖包）。
> 请不要输出任何不相关的寒暄或废话。”**

**原理解析**：
格式化是给自动化脚本或强迫症开发者的福音。
大模型很喜欢在开头说“好的！很乐意为您解答”，在结尾说“希望这个答案对您有帮助，如果还有其他问题...”。通过 Format 约束，你可以强制它变成一个冰冷高效的接口。对于你在代码里用 API 调用 AI 时，这种 Markdown 格式的绝对规范，是防止后续解析报错的根本依托。

---

## 小结示范

**反面教材（灾难般的效果）：**
❌ “帮我写个扣库存的代码，老是超卖很烦。”

**RCTF 完全体（绝佳的效果）：**
✅ “**[Role]** 你是一位高级 Java 并发专家。**[Context]** 我在开发基于 Spring Boot 和 Redis 的医院抢号系统，早 8 点峰值很高，常发生库存超卖。**[Task]** 请帮我编写一段通过 Redis 执行 Lua 脚本来保证原子性扣减库存的完整方法代码。**[Format]** 请直接输出可直接粘贴运行的 Service 层方法代码，包含所需 imports，并在复杂逻辑处添加中文单行注释。不要输出多余的解释与寒暄。”

掌握了基础的 RCTF 后，你已经能打败 90% 不会用 AI 的人了。但如果是极为复杂的疑难杂症，仅靠 RCTF 可能还不够。请看下一章：**[进阶 Prompt 策略 (CoT/Few-Shot)](./advanced-prompting)**。
