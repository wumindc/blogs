# Prompt 核心框架：角色、上下文、任务与格式

> Prompt 被讲得最玄乎的时候，往往也是它最容易被误解的时候。很多人一听“提示词工程”，脑子里先冒出来的是措辞技巧，像在练一种面向模型的话术。
> 可真实工程里，Prompt 真正值钱的部分通常不在“说得漂亮”，而在“边界划得清楚、上下文喂得合适、输出结构接得住”。

::: info 这篇文章重点
- 一个实用 Prompt 应该至少包含哪些部分
- 为什么 system / user / context 的分层很重要
- 如何用格式约束和失败兜底提高稳定性
:::

## 1. 一个好 Prompt 最少要回答四个问题

你可以把基础 Prompt 组织成四部分：

| 维度 | 要回答的问题 |
| --- | --- |
| 角色（Role） | 你希望模型站在什么职责位置上工作 |
| 上下文（Context） | 它需要知道哪些背景信息 |
| 任务（Task） | 它具体要做什么 |
| 格式（Format） | 结果应该长什么样 |

很多不稳定输出，往往是因为这四部分缺了一两块。

## 2. system 提示词的价值，是定义边界

system 提示词最适合放：

- 角色定义
- 全局约束
- 安全边界
- 输出原则

例如：

```text
你是企业内部知识助手。
如果证据不足，请明确说明不知道。
不得编造来源。
优先输出结构化要点。
```

这比在每轮用户输入里反复强调更稳定。

## 3. 上下文要尽量相关，而不是尽量多

模型并不会因为你塞了更多背景就自动更懂。相反，过长、过杂的上下文常常会降低输出质量。

更好的原则是：

- 只提供决策所需信息
- 保持信息来源清晰
- 对长资料先做压缩或检索

## 4. 格式约束是工程稳定性的核心

如果输出需要被系统继续消费，就应该尽量明确格式，例如：

- JSON
- Markdown 列表
- 固定字段模板

相比“请输出详细分析”，更稳的写法往往是：

```text
请按以下 JSON 字段返回：
summary、risk_level、evidence、next_action
```

## 5. 失败兜底也要写进 Prompt

很多系统默认模型“一定能答”。更稳妥的方式是提前告诉它：

- 证据不足怎么办
- 字段缺失怎么办
- 多种情况冲突怎么办

这类兜底规则对稳定性帮助很大。

## 小结

Prompt 的基础功，不在辞藻，而在结构。把角色、上下文、任务和格式组织清楚，再辅以适度的失败兜底，往往比盲目“润色提示词”更有效。
