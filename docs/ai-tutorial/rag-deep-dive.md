## 1. Chunking：防止语义碎裂的文本外科手术切割法

如果在往向量库里极其简单粗暴地按“每 500 个字切一刀”写入：
* 原句是：“王总宣布（此处字数满500遭腰斩切断落入块1）。。。”然后“。。。（块2起始开头）今年年报利润下调”。
* 结果：这两块碎片进入高维向量空间计算后会被极度分离散落天涯且失去原主语境被极具扯断孤立导致召回引擎极其严重失败迷航。

### 滑动窗口与父子溯源提取 (Parent Document Retriever)
高级的高阶数据流入仓架构设计师会在这层重重构布下这套极其缜密的切剥隔离方法：

1. **滑块重叠连贯补丁 (Overlap Strategy)**：
   绝不采用生切。假设按块 `Chunk = 512 Tokens` 切刀。我们会强行要求在刀口上进行由于上一个块与后一个新块设定如 `Overlap = 50 tokens` 字眼重混纠结粘连冗余的搭接补全长挂，强行保护上下的那微残微断骨节语义牵引的连贯不断！
2. **意群断隔分级切切法 (Semantic Chunking 语义切刀法)**：
   不再依赖瞎眼的计算切长度！通过 Python 或者 NLP 工具利用检测标点例如句号，感叹极其深换行甚至是 Markdown 特有的标题头 `##` 为首准刀界来保证在绝断由于长一段被完整语义包裹的长篇。
3. **父大子碎追溯唤召唤术 (Small-to-Big Retrieval 法术)**：
   这是企业级提高极效且极绝不会因为大内容塞爆大背景且又不丢失极极全上下文这极其矛盾终极极点核心极其强悍由于大极招引流策略！
   **写入时**：我们将一整页巨大由于 2000 字非常极其由于巨大的由于章节叫做父大宏文件 (`Parent_Doc`)。并在写入并落盘时候再把它狠狠极碎刀细割切片由于几十段极其小的极其只有一百几十子的极其细针由于细小子片段被落库。
   **查询时**：用户一句短问题查询。这由于由于极其细针般极其极其微末的细小特征被向量表计算比对命撞极其由于命极其死极其极其精准精准撞极击穿由于找到某个子片短落！在准备由于召回发外的时候！**我们不要这个召出撞子片！它只作为钩子**。用它顺挂着那个由于勾找回当时原本那个极大极宏大包含全境上下文背景极其那一大片全部整块 `Parent_Doc` 大整长文全部捞拉提出来。

## 2. RRF 与多路找寻：混合检索架构（Hybrid Search）的双螺旋

前面讲了由于密集稠密型的高多维抽象向量（Dense Vector）搜寻强悍，那为还要因为提什么 BM25？

**致命弱点场景**：如果企业的资料库是有着数以万记的各种有着如 “商品代码 `SKU-TX#992-B`” 或员工极其生僻不具备连词义姓名这极种极其专用并且独特专有名词！基于大模型训练出来的向量化神经模型如果当时大训的时候从未见闻根本不认识不知道，这极其由于它极会在词汇搜索迷失！

**架构级应对大招 —— RAG 3.0 的基石：多路强力融合并进 (Hybrid Retrieval)**

网关在接收到提问的第一秒瞬间兵分两极路同时爆开进大并发流库进行追发大打。
1. **语义抽象通道 (Vector DB Milvus)**：拿长高维向量去寻那种“与此情此景类似极度大意思”的大面文。
2. **死板精准刺刀通道 (ElasticSearch BM25)**：用字根搜索找出孤僻罕见的名词，这对于极其极其专精名字有极大极查出！

**RRF (倒数排序集极合算法) 进行终极融合**：
它是通过各自的单独在极其独立大表排名中占据位置（Rank），利用数学重构排序表 `score = 1 / (k + rank)` 将两路强强极其融合！

## 3. Rerank 重排精筛模型：压轴的昂贵大算力提纯拦截网

上述前端过滤手段难免粗糙。如果你把初筛带来的 40 篇长文极其直接全部喂进主 LLM，极其会因为极其极大极大噪音触发极其严重 **Lost in the Middle（中间注意力遗失）** 幻觉！

**引入交叉重编码极大高算精细审判模型（Cross-Encoder Reranker）**。

在网关大发去终极 LLM 前：
把用户原查询 `Query` 和初查挑出那前 40 篇长文档，每一对因为都是组合极极其细微塞进入这个拥有参数（比如 BGE-Reranker）的专门打分大神经模型里进行极其深层交叉推理！！极其大极其算力极其重！！它非常慢所以只能算前几十篇！！
经极其这一由于大深考细打这分重压锤并打出的极挑分并且排在仅仅由于最高且第一、二名极其文章将真正送去生成解答大由于终极其推理大系统。
