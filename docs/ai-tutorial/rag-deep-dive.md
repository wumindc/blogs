# 深入 RAG 水深区：文档切块与多路重排

> 在理论上，用向量检索（Embedding）匹配找文档的方法美妙绝伦。然而一旦在企业业务中正式落地，你会绝望地发现一个残酷的工程现实：**用最基础的向量匹配搜出来的东西，命中率简直惨不忍睹。**

比如，一篇五万字的关于《2026 年新能源汽车出海战略规划》报告的 PDF。如果你把它一整个塞进大模型去算成 1 个向量坐标，它在几千个维度的空间里的语义信息会被互相稀释成一团乱麻。遇到具体关于某个车型“零跑 C11” 电池数据询问时，这个宏大的总结向量坐标与极其微小具体的提问坐标差了十万八千里，根本召回（捞取回来）不到这段内容。

要解决这个工程痛点，我们就进入了高阶 RAG（检索增强生成）必不可免的三个脏活累活。

---

## 1. 源头治理第一刀：文段查验碎尸法 (Chunking)

为了防止向量语义稀释，我们绝不能把整篇文章直接转成向量存入库。我们要在落库前，把这篇文章像切肉卷一样，切成具有固定大小（如 500 个字或 1000 个字）的词块，称为 **Chunk**。
只有当块切得足够小，针对这一块去计算出的唯一的一个向量坐标才能高度凝练代表这小小段落的核心语义。

但是怎么切？
*   **按字数硬切**：最蠢的做法。每 500 个字符一挥刀。这很容易在句子中央截断，导致 `“苹果公”` 和 `“司本季收益...”` 被硬生生分在两块。当用户搜索“苹果公司”时，这两块因为语义破损不仅凑不到一起，反而找不到。
*   **由于滑窗引发的堆叠 (Overlap)**：目前主流做法。切下一个 500 字的 chunk 时，强制往前回退 100 个字作为“重叠区 (Overlap)”进行衔接。这样能最大限度防止边缘断句处的知识遗漏。
*   **高阶解析 (Markdown/目录切分)**：大厂更倾向于不按字符切，而是去解析你那份极其难搞的 PDF 内部结构的标题（H1，H2，H3）。确保标题、段落和表格被当作一个整体逻辑结构切分存入。

## 2. 大道至简的防抖护甲：多路混合召回 (Hybrid Search)

当我们沉迷于使用时髦的“向量检索（Embedding 语义找相似）”时，你会悲哀地发现向量有一个天生的致命缺陷——**它对极其严苛的精准词汇极其不敏感**。

比如，用户输入：`“查询订单号 O-2026-XDF932 的交易状态”`。
由于这是一个随机生成的极其鬼畜的单号，它没有任何“语义可言”，大模型计算它在多维空间中的向量位置时完全懵逼。向量相似度检索在这个精准长尾请求上形同废纸。

为了兜底这种需求，业内所有成熟的 RAG 会放弃对新技术的绝对迷信，转而使用 **混合召回 (Hybrid Search)** 策略：
同一句话，兵分两路：
*   **第一路：传统关键字倒排索引 (例如 BM25 / ES)**。依靠字面上把 `O-2026-XDF932` 匹配得毫厘不差捞出两篇文档。
*   **第二路：高阶向量神经嵌入 (Vector Search)**。基于用户对业务口语化描述“如何解决充值报错”捞出两篇文章。

两路汇总，一共获得包含关键词准确性和语义广泛性的 4 篇候选文档池。

---

## 3. 王者裁决器：重排序 (Rerank)

通过两路甚至更多路的漏斗捞出来可能十几篇文档，但这时候面临一个巨大的考验。
由于这十几篇长文档加在一起长度可能突破了两三万字，一方面把它们全部塞给大模型的 Context 窗口导致 API 费用暴涨，另一方面我们上一章提过的长文本 **“海里捞针”** 效应会导致大模型迷失在信息的垃圾堆中。

我们必须精简，**只要那绝对最精华的 Top 3 段落。**

于是我们引入最终级的守门员网络模型，统称为 **Reranker (重排器)** 专门负责重新审视得分。
*   有别于刚才计算高维坐标夹角这种极其轻量的计算，Reranker 模型会对用户的“问题”和刚才“粗捞上来的 15 篇备选文档”，进行字对字级别的极其变态的**深度交叉注意力运算 (Cross-Attention)**。
*   **这就好比从泛泛而谈的“盲选相亲（向量初筛）”，变成了精确到毛孔级别的“面对面核验（重排复审）”。**
*   这虽然会消耗大量的计算用时，但由于此时的基数已经从几十万篇缩小到了 10 篇文档，时间可以接受。

重排序完成后，只把得分压倒性最高的 Top 3 扔进那段珍贵的 Prompt 当中喂给大模型做最后的阅读理解。
