# 解密“模型”与“参数”：从权重到量化部署

> 很多人第一次下载模型权重时都会有点错愕：代码没多少，真正占地方的是那一大堆你乍看完全看不懂的数值文件。
> 也就是从这一步开始，所谓“模型”这件事会突然变得非常具体。它不再只是一个抽象能力名词，而是一份会占磁盘、吃显存、决定部署成本的真实工程对象。

::: info 这篇文章重点
- 参数究竟是什么
- 参数量为什么会和存储、显存直接相关
- FP16、BF16、INT8、INT4 等精度为什么重要
- 量化适合什么场景，不适合什么场景
:::

## 1. 模型在工程上首先是一组权重

从工程角度看，模型不是“写满业务规则的代码”，而是一组巨大的参数权重。

这些参数本质上是：

- 矩阵
- 向量
- 层与层之间的数值连接关系

也就是说，模型的“能力”并不是写在 `if-else` 里，而是分布在海量数值之中。

## 2. 为什么参数量直接影响部署成本

参数量越大，通常意味着：

- 模型文件越大
- 加载所需内存或显存越高
- 推理时带宽压力更大

这也是为什么“7B”和“70B”不只是抽象数字，而会直接决定：

- 单卡能不能装下
- 服务要不要分片
- 成本大概落在哪个量级

## 3. 精度为什么重要

模型权重需要用某种数值精度存储，例如：

- FP16
- BF16
- INT8
- INT4

精度越高，通常保真度越好，但占用资源也越多；精度越低，存储和推理成本下降，但可能带来能力损失。

## 4. 量化在解决什么问题

量化的核心目标是：**用更少的存储和显存，承载尽量接近原模型的能力**。

它最直接的收益包括：

- 降低显存占用
- 提升部署可行性
- 在一定条件下改善吞吐和成本

但量化不是零代价优化。常见代价是：

- 精度下降
- 某些复杂推理任务更容易退化
- 不同量化方法效果差异明显

## 5. 一个实用判断

如果你的场景是：

- 信息抽取
- 分类
- 检索辅助
- 流水线中间步骤

更激进的量化通常更容易接受。

如果你的场景是：

- 复杂推理
- 高质量生成
- 代码或数学任务

就应该更谨慎评估量化带来的精度损失。

## 小结

理解“模型”最重要的一点是：它本质上是一组庞大的数值权重。参数量、精度和量化方式，会直接影响部署可行性、成本和质量边界。
