# 什么是 Token 与上下文窗口：从 BPE 到成本控制

> Token 这个词看上去像个基础概念，偏偏又是最容易把项目预算、延迟和上下文体验一起搅乱的那个基础概念。
> 很多团队第一次接入模型后，最先感受到的甚至不是“模型很聪明”，而是“怎么刚聊几轮，账单和长度限制就都开始给脸色看了”。真到这时，Token 就不再是术语，而是系统设计问题了。

::: info 这篇文章重点
- Token 和字符、单词有什么区别
- BPE 为什么会成为常见分词方式
- 为什么上下文窗口会直接影响成本和效果
:::

## 1. Token 不是“一个字”也不是“一个单词”

模型看到的输入并不是原始自然语言，而是被切分后的 Token 序列。

一个 Token 可能是：

- 一个汉字
- 一个英文单词的一部分
- 一个标点
- 一段常见字符串片段

所以字符数、单词数和 Token 数并不等价。

## 2. BPE 在解决什么问题

像 BPE（Byte Pair Encoding）这类方法的目标，是在“切得太碎”和“词表太大”之间找到平衡。

它的价值在于：

- 常见片段可以合并，提高效率
- 生僻词仍可拆分，避免完全 OOV

这使得模型能更灵活地处理多语言和长尾表达。

## 3. 为什么 Token 直接影响成本

大多数模型服务的计费、延迟和上下文限制都与 Token 数有关。Token 越多，通常意味着：

- 输入更贵
- 输出更贵
- 延迟更高
- 上下文更容易被占满

所以在工程里，控制 Token 使用量不仅是省钱，也是在提高系统稳定性。

## 4. 上下文窗口是什么

上下文窗口可以理解为模型一次能“看到”的 Token 上限。它不是说模型真正理解了全部内容，而是：

- 超过这个长度通常无法一次完整输入
- 越接近上限，成本和注意力负担越大

因此长上下文并不等于随便塞。真正重要的是把最相关的信息放进去。

## 5. 工程上怎么控制 Token 成本

常见做法包括：

- 截断无关历史对话
- 对长文先摘要再送模型
- 检索时只组装高相关内容
- 在网关层估算和限制 Token

## 小结

Token 是模型处理语言的基本单位。理解它，能帮助你更准确地判断：

- 输入为什么这么长
- 成本为什么会上涨
- 上下文为什么会成为系统设计重点
